### FB-004 — 다음 해금까지 남은 거리가 매일 체크에서 불투명

| 필드 | 값 |
|------|-----|
| 날짜 | 2026-04-19 |
| 채널 | 본인 검증 |
| 분류 | UX / 동기부여 |
| 우선순위 | **P2** |
| 상태 | **반영완료** (2026-04-24, v1.2 싹 성장 시스템 — 코드 반영) |

**내용**
- 매일 체크할 때 다음 보상 해금까지 얼마나 남았는지 힌트가 없음
- 유저는 해금 토스트가 뜨기 전까지 자신이 보상에 가까워지고 있다는 감각을 얻지 못함

**근본 원인 해석**
- 해금 시스템은 구현돼 있으나, 임박 신호가 없어 **"가변 보상 스케줄" 효과가 해금 순간에만 제한적으로 발생**
- 그러나 숫자 기반 진행도(예: "5일 남음")는 다크패턴·외재적 동기 과잉·룰 10(존재 이유) 침해 위험

**해결 방향 — 방향 1 채택 (싹 성장 시각화 내장)**
- 숫자 없음. **싹이 자란 단계 자체가 진행도**
- 해금 임박 시 시각적 티저 (예: 꽃봉오리 직전 상태)
- 노이즈 0, 브랜드 완벽 통합, 다크패턴 회피

**결정 근거**
- 헌법 2(5초 UX), 10(차별점), 8(본인 매일 쓰는가) 모두 통과
- FB-002(싹 성장 축하) + FB-003(해금 안내) + FB-004(진행도) 세 피드백이 **하나의 "싹 성장 시스템"**으로 수렴

**관련 파일 (v1.2 작업 대상)**
- `src/components/CelebrationOverlay.tsx` — 컨페티 → 싹 성장 애니메이션
- `src/components/UnlockToast.tsx` — 성장 이벤트 시각 언어로 리디자인
- `src/constants/rewards.ts` — 성장 단계와 해금 티어 매핑
- `src/store/habitStore.ts` 또는 신규 `growthStore.ts` — 습관별 성장 상태 저장
- `phase4-plan.md` §3 — v1.2 스트릭 보상 시스템 섹션 통합 반영

### FB-003 — 해금 시스템의 존재가 유저에게 사전 안내되지 않음

| 필드 | 값 |
|------|-----|
| 날짜 | 2026-04-19 |
| 채널 | 본인 검증 |
| 분류 | UX / 동기부여 |
| 우선순위 | **P2** |
| 상태 | **반영완료** (2026-04-24, v1.2 싹 성장 시스템 — 임박 힌트로 사전 신호 제공) |

**내용**
- 해금 시스템(`rewards.ts`, `UnlockToast`)은 이미 구현·연동 완료
- 그러나 유저는 7일 흐름 도달 전까지 해금의 존재를 전혀 모름 → 첫 주 동기부여 공백

**근본 원인 해석**
- 온보딩·설정·메인 어디에도 해금 힌트가 없음
- "우연히 발견"하는 구조 → 기대감 기반 동기부여 효과 절반 이상 소실

**해결 방향**
- v1.2 싹 성장 시스템 도입 시 자연 해결:
  - 성장 단계 자체가 "뭔가 자라고 있다"는 시각 신호
  - 유저는 보상 존재를 **명시적으로 안내받지 않아도** 직관으로 인지
- 단독 해결 시도 금지 (예: "7일 달성 시 해금!" 배너 추가) → 룰 6 위반 우려

**관련 파일**
- FB-004와 동일 (통합 작업)

### FB-002 — 성공 이미지가 브랜드와 동떨어짐

| 필드 | 값 |
|------|-----|
| 날짜 | 2026-04-19 |
| 채널 | 본인 검증 |
| 분류 | UX / 브랜딩 |
| 우선순위 | **P1** |
| 상태 | **반영완료** (2026-04-24, v1.2 — 컨페티 제거, 싹 성장 시각으로 교체) |

**내용**
- 현재 체크 완료 시 컨페티(8색, 브랜드 외 색상) + 랜덤 메시지 6종 표시
- "앱 이름 싹 / 태그라인 '작은 습관이 자라는 곳'"과 축하 화면이 무관
- 유저가 심은 **"내 습관(싹)"과 성공 화면이 연결되지 않음** → 감정이입 없음

**근본 원인 해석**
- 앱 전체가 성장 메타포("싹", "씨앗이 자라려면…")인데 보상 순간만 일반적 파티 연출
- "물 주기(체크) → 성장(축하) → 수확(해금)" 서사로 묶이지 않음

**해결 방향**
- 컨페티 제거 → 세이지 그린 톤의 싹 성장 애니메이션으로 교체
- 매일 체크 = 싹 성장 +1 단계 (시각적 진행도)
- 해금 순간 = 큰 성장 이벤트 (UnlockToast 리디자인)

**범위 판단**
- v1.0.1 단독 교체(옵션 A)도 가능했으나, FB-003·FB-004와 구조적으로 같은 레이어라 **v1.2에서 통합 설계**로 합의
- v1.0.1 기간은 홍보·관측·본인 검증에 집중 (룰 4 준수)

**관련 파일**
- FB-004와 동일 (통합 작업)

### FB-001 — 알람 없으면 앱 존재 자각이 없음

| 필드 | 값 |
|------|-----|
| 날짜 | 2026-04-19 |
| 채널 | 본인 검증 (출시 3일차) |
| 분류 | UX / 리텐션 |
| 우선순위 | **P1** |
| 상태 | **반영완료** (2026-04-19, 온보딩 4단계화 — 코드 반영) |

**구현 요약**
- `app/onboarding.tsx`: 3단계 → 4단계 (환영 → 습관 선택 → 알람 설정 신규 → 사용 가이드)
- `src/utils/defaultReminderTime.ts` (신규): 습관 이름 키워드 기반 기본 시간 매칭
- `src/components/TimePicker.tsx`: `renderTrigger` 커스텀 trigger prop 지원 (호환 유지)
- 카피: "씨앗이 자라려면 매일 같은 시간 물을 줘야 해요 / 알람이 그 시간을 대신 기억해드릴게요"
- 건너뛰기는 존중, 묵시적 허용. 선택된 습관의 reminderTime은 null 처리 후 진행

**내용**
- 알람을 설정하지 않으면 앱을 사용 중이라는 자각 자체가 사라짐
- 온보딩에서 알람 설정을 능동적으로 유도하거나, 위젯을 넘어 잠금화면까지 노출 단계를 끌어올려야 함

**근본 원인 해석**
- 유저의 능동적 "앱 열기" 의지에만 기대는 구조는 리텐션 구조 결함
- 우리가 유저를 찾으러 가는 장치(알람·위젯·잠금화면)가 필요

**해결 방향**
- **단기 (v1.0.1 내)**: 온보딩을 3단계 → 4단계로 확장. "습관 선택" 직후 "알람 설정" 단계 삽입. 알람이 왜 필요한지 설득 카피 포함
- **중장기 (신호 누적 시)**: 잠금화면 위젯은 로드맵상 v2.0+ 항목. Week 2 외부 피드백에서 동일 패턴 빈도 높으면 v1.2로 승격 검토

**관련 파일**
- `app/onboarding.tsx` (4단계화 대상)
- `phase4-plan.md` (잠금화면 위젯 승격 여부 기록처)
