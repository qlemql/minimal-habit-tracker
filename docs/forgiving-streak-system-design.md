## 1. 학술적 근거

### 1-1. Phillippa Lally의 습관 형성 연구 (UCL, 2009)

University College London의 Phillippa Lally 연구팀이 96명을 대상으로 수행한 연구의 핵심 발견:

| 발견 | 내용 |
|------|------|
| **평균 습관 형성 기간** | 66일 (범위: 18일~254일) |
| **하루 빠짐의 영향** | 자동화 점수 평균 0.29점 하락 (무시해도 될 수준) |
| **핵심 결론** | 가끔 하루를 빠지는 것은 습관 형성 과정에 심각한 영향을 주지 않는다 |

> 원문: "Missing the occasional opportunity to perform the behaviour did not seriously impair the habit formation process: automaticity gains soon resumed after one missed performance."
>
> — Lally et al., European Journal of Social Psychology, 2010

### 1-2. 스트릭 심리학의 함정

| 연구 결과 | 출처 |
|-----------|------|
| 30일 스트릭을 잃는 고통 = 스트릭을 쌓는 즐거움의 **2배** | 2019년 행동심리학 연구 |
| 스트릭 기반 추적 사용자는 하루 빠짐 후 습관을 **완전히 포기할 확률 63% 더 높음** | 2020년 Journal 연구 |
| "What-the-hell effect" — 한 번 실패하면 "망했으니 그냥 포기하자"가 됨 | University of London |

### 1-3. "이틀 규칙" (Two-Day Rule)

Matt D'Avella가 대중화한 개념 (YouTube 조회수 400만+):

> **"하루는 괜찮다. 이틀 연속은 안 된다."**

- 하루 빠짐 = 일시적 흔들림 (blip)
- 이틀 연속 빠짐 = 새로운 패턴의 시작
- 이 규칙을 적용한 참가자들은 전통적 스트릭 대비 **37% 더 오래** 습관 유지 (British Journal of Health Psychology, 2021)
- 24시간 내 재개한 사람은 장기 일관성 유지 확률 **71% 더 높음**

### 1-4. 설계 원칙 도출

```
과학이 말하는 것:
  하루 빠짐 → 습관 형성에 거의 영향 없음

기존 앱이 하는 것:
  하루 빠짐 → 스트릭 0으로 리셋 → 사용자 좌절 → 앱 삭제

MHT가 해야 하는 것:
  하루 빠짐 → 과학에 맞게 반응 → "괜찮아, 이어가자" → 사용자 유지
```

|------|------|
| **메커니즘** | 가상 화폐(200젬)로 "스트릭 프리즈" 아이템 구매, 최대 2개 보유 |
| **작동** | 하루 빠지면 자동 사용, 스트릭 유지 |
| **장점** | 직관적, 게이미피케이션과 잘 어울림 |
| **단점** | "아이템을 써야 보호된다" = 빠지는 것이 기본적으로 나쁘다는 전제, 사전 구매 필요 |
| **MHT 적용 가능성** | **낮음** — 게이미피케이션 시스템이 없고, "프리즈"라는 단어 자체가 부정적 |

### 2-2. Loop Habit Tracker — 습관 강도 (Exponential Smoothing)

| 항목 | 내용 |
|------|------|
| **메커니즘** | 지수평활법으로 습관 점수 계산. 수행하면 올라가고 빠지면 내려가되, 완전히 0이 되지 않음 |
| **공식** | `완료일: S(n) = (1-α)S(n-1) + α` / `미완료일: S(n) = (1-β)S(n-1)` |
| **장점** | 수학적으로 공정, 장기 노력을 인정, 하루 빠져도 점수가 조금만 떨어짐 |
| **단점** | 사용자가 공식을 이해하기 어려움, 시각적 표현이 직관적이지 않음, "점수"가 또 다른 부담 |
| **MHT 적용 가능성** | **중간** — 내부 로직으로는 좋지만 사용자에게 보여주는 방식이 과제 |

### 제안 A: "이어가기" (권장안)

> **한 줄 요약: 하루 쉬어도 흐름은 끊기지 않는다.**

#### 메커니즘

```
핵심 규칙: "이틀 연속 빠지지 않으면 흐름이 이어진다"

Day 1: ✓ 완료     → 흐름 1일
Day 2: ✓ 완료     → 흐름 2일
Day 3: — 빠짐     → 흐름 2일 (유지, 쉼표 표시)
Day 4: ✓ 완료     → 흐름 3일
Day 5: ✓ 완료     → 흐름 4일
Day 6: — 빠짐     → 흐름 4일 (유지)
Day 7: — 빠짐     → 흐름 끊김 → 새로운 시작
Day 8: ✓ 완료     → 흐름 1일 (다시 시작)
```

#### 카운팅 방식

- **흐름 일수** = 해당 기간 내 실제 수행한 날의 수 (빠진 날은 카운트하지 않지만 흐름을 끊지도 않음)
- **쉼표(,)** = 하루 빠짐의 시각적 표현. 쉼표가 있어도 문장은 이어진다는 비유
- **마침표(.)** = 이틀 연속 빠짐. 문장이 끝남. 새로운 문장 시작

#### 시각적 표현

```
주간 뷰 (메인 화면):

  월  화  수  목  금  토  일
  ●   ●   ○   ●   ●   ●   ○
         쉼             쉼

  ● = 완료 (습관 색상, 채워진 원)
  ○ = 쉼표 (습관 색상, 빈 원, 점선 테두리)
  × = 끊김 (회색, 작은 점)

  "흐름 5일째" (●의 개수만 카운트)
```

```
메인 화면 스트릭 표시:

  ┌─────────────────────────┐
  │  🏃 운동하기             │
  │                         │
  │  흐름 23일째             │
  │  ○●●○●●●               │
  │                         │
  └─────────────────────────┘

  * ○는 쉼표(쉰 날), ●는 수행한 날
  * 최근 7일을 미니 도트로 표시
```

#### 감정적 임팩트

| 상황 | 기존 앱의 반응 | MHT "이어가기"의 반응 |
|------|---------------|---------------------|
| 하루 빠짐 | "스트릭 0. 다시 시작하세요" | "쉬어갔네요. 흐름은 계속돼요" |
| 이틀 빠짐 | "스트릭 0. 다시 시작하세요" | "새로운 시작이에요. 이전 기록은 남아있어요" |
| 빠진 다음날 돌아옴 | 아무 메시지 없음 | "다시 왔네요! 흐름이 이어집니다 ✓" |

#### 엣지 케이스

| 상황 | 처리 방식 |
|------|----------|
| 1일 빠짐 | 흐름 유지, 쉼표 표시 |
| 2일 연속 빠짐 | 흐름 끊김, 새로운 흐름 시작 (이전 흐름 기록 보존) |
| 3일+ 연속 빠짐 | 흐름 끊김, 돌아오면 "다시 시작" (비난 없이) |
| 1주일 빠짐 | 흐름 끊김, 돌아오면 "돌아온 걸 환영해요" 메시지 |
| 주 3회 습관 | 해당 없음 — MHT는 매일 습관만 지원 (Phase 1) |

#### 3개 습관 제약과의 연계

- 각 습관이 독립적인 "흐름"을 가짐
- 3개 모두 같은 날 완료하면 **"오늘의 흐름 완성"** 표시
- 3개 중 2개만 해도 한 습관의 흐름이 끊기지는 않음 (각각 독립)

#### 데이터 구조 변경

```typescript

  habitId: string
  date: string        // YYYY-MM-DD

// 변경 없음 — 이어가기 로직은 HabitLog 데이터로 계산 가능

  currentFlowStartDate: string // 현재 흐름 시작일
  hasBreathingDay: boolean     // 오늘이 쉼표 상태인지
  longestFlow: number          // 최장 흐름 기록
} {
  // 오늘부터 역순으로 탐색
  // 연속 2일 미완료를 찾으면 흐름 시작점
  // 그 사이의 완료 일수만 카운트
}
```

### 제안 B: "습관 온도" (대안 1)

> **한 줄 요약: 습관은 온도처럼 서서히 올라가고 서서히 식는다.**

#### 메커니즘

```
습관 수행 → 온도가 올라감 (+5~10도)
습관 빠짐 → 온도가 내려감 (-3도/일)
온도 범위: 0°~100°

온도 구간:
  0°~20°   = 씨앗 (시작 단계)
  21°~50°  = 새싹 (형성 중)
  51°~80°  = 성장 (안정화 단계)
  81°~100° = 뿌리내림 (습관 완성)
```

#### 시각적 표현

```
메인 화면:

  ┌─────────────────────────┐
  │  🏃 운동하기             │
  │                         │
  │  ████████░░  72°        │
  │  성장 단계               │
  │                         │
  └─────────────────────────┘

  * 온도 바가 습관 색상으로 채워짐
  * 어제보다 올랐으면 미세한 glow 효과
  * 어제보다 내렸으면 살짝 어두워짐 (비난X, 시각적 변화만)
```

#### 감정적 임팩트

| 장점 | 단점 |
|------|------|
| 완전 리셋이 없어 심리적 안전감 | "72도"가 직관적이지 않을 수 있음 |
| 올라가는 느낌이 동기부여 | "온도"라는 비유가 습관과 자연스럽게 연결되지 않을 수 있음 |
| 하루 빠져도 3도만 빠짐 | 숫자에 민감한 사용자에게 또 다른 부담 |

#### 엣지 케이스

| 상황 | 처리 방식 |
|------|----------|
| 1일 빠짐 | -3도 (72° → 69°). 거의 변화 없음 |
| 2일 연속 빠짐 | -6도 (72° → 66°). 눈에 띄지만 크지 않음 |
| 1주일 빠짐 | -21도 (72° → 51°). 구간은 유지될 수 있음 |
| 0도 도달 | "다시 시작할 준비가 됐을 때 탭하세요" |

#### 이 제안의 한계

- "5초 원칙"에 약간 부담: 온도, 구간, 변화량을 이해해야 함
- Loop의 점수 시스템과 유사한 단점 공유 (숫자가 또 다른 부담)
- 미니멀 철학과 약간 충돌 (정보량이 늘어남)

### 제안 C: "씨앗 키우기" (대안 2)

> **한 줄 요약: 습관은 씨앗에서 나무로 자란다. 하루 물을 안 줘도 나무는 죽지 않는다.**

#### 메커니즘

```
습관의 성장 단계:

  1단계: 씨앗     (1~7일)     — 매일 필요, 연약한 시기
  2단계: 새싹     (8~21일)    — 하루 빠져도 괜찮아지기 시작
  3단계: 묘목     (22~44일)   — 이틀 빠져도 흐름 유지
  4단계: 나무     (45~66일)   — 삼일 빠져도 흐름 유지
  5단계: 큰 나무  (67일+)     — 습관이 뿌리내림

즉, 습관이 성장할수록 "관용 일수"가 늘어남
```

#### 시각적 표현

```
메인 화면:

  ┌─────────────────────────┐
  │  🏃 운동하기             │
  │                         │
  │      🌱                  │
  │   새싹 · 14일째          │
  │                         │
  └─────────────────────────┘

성장 단계별 아이콘:
  🌰 씨앗 → 🌱 새싹 → 🪴 묘목 → 🌳 나무 → 🌲 큰 나무
```

#### 감정적 임팩트

| 장점 | 단점 |
|------|------|
| 성장 비유가 직관적이고 따뜻함 | 단계 시스템이 "5초"에 이해되지 않을 수 있음 |
| "나무는 하루 안 줘도 안 죽어"가 자연스러움 | 단계별 관용 일수가 다른 것이 혼란 유발 가능 |
| 성장 자체가 보상감 | 이모지/아이콘 의존도가 높음 |

#### 이 제안의 한계

- 5단계 시스템 = 설명이 필요함 = 미니멀과 충돌
- "지금 내가 몇 단계인지"를 신경 쓰게 됨 = 새로운 부담
- 단계별 관용 일수 차이를 사용자가 기억해야 함

## 4. 권장안: "이어가기" 시스템 (제안 A)

### 4-1. 왜 이어가기인가

| 평가 기준 | 이어가기 (A) | 습관 온도 (B) | 씨앗 키우기 (C) |
|-----------|:-----------:|:------------:|:--------------:|
| 5초 이해 가능 | **O** | △ | X |
| 추가 UI 불필요 | **O** | X (바 필요) | X (아이콘 필요) |
| 데이터 구조 변경 | **없음** | 있음 (온도 저장) | 있음 (단계 저장) |
| 학술 근거 부합 | **O** (Two-Day Rule) | O (지수평활) | △ (비유적) |
| 감정적 안전감 | **높음** | 중간 | 높음 |
| 미니멀 철학 부합 | **최고** | 중간 | 낮음 |
| 서버 불필요 | **O** | O | O |

### 4-2. 이어가기의 핵심 장점

1. **기존 데이터 구조 변경 없음** — HabitLog만으로 계산 가능
2. **설명이 한 문장** — "하루 쉬어도 흐름은 이어져요"
3. **학술적으로 정확** — Lally 연구 + Two-Day Rule과 완벽히 일치
4. **시각적으로 단순** — 기존 주간 뷰에 ○(쉼표)만 추가
5. **부정적 언어 없음** — "프리즈" "리셋" "실패" 대신 "쉼표" "이어가기"

| **이어가기** | 흐름을 이어간다 | **최종 선정** — 가장 직관적, 긍정적, 한국어 자연스러움 |
| 쉼표 | 멈춤이 아닌 잠깐의 쉼 | 부제로 활용 가능 |
| 흐름 | 스트릭의 한국어 대체 | 내부 용어로 사용 |
| 여유 | 여유를 허용하는 시스템 | 너무 느슨한 느낌 |
| 숨고르기 | 잠시 숨을 고르고 이어감 | 아름답지만 약간 길다 |

### 5-2. 용어 체계

| 기존 (영어/업계) | MHT 용어 (한국어) | 의미 |
|-----------------|-------------------|------|
| Streak | **흐름** | 연속 수행 기간 |
| Streak count | **흐름 N일째** | 현재 흐름의 수행 일수 |

## 7. 알고리즘 상세

### 7-1. 흐름 계산 로직

```typescript
interface FlowResult {
  currentFlowDays: number;       // 현재 흐름에서 수행한 일수
  currentFlowStartDate: string;  // 현재 흐름의 시작일 (YYYY-MM-DD)
  isBreathingToday: boolean;     // 오늘이 쉼표 상태인지
  longestFlow: number;           // 역대 최장 흐름
  status: 'active' | 'breathing' | 'broken';
}

function calculateFlow(
  logs: HabitLog[],
  today: string
): FlowResult {
  // 1. 오늘부터 과거로 탐색
  // 2. 각 날짜를 확인:
  //    - 완료 → 수행 일수 +1, 계속 탐색
  //    - 미완료 1일 → 쉼표, 수행 일수 변화 없음, 계속 탐색
  //    - 미완료 2일 연속 → 흐름 끊김, 탐색 종료
  // 3. 결과 반환

  const sortedDates = getSortedDatesDescending(logs, today);
  let flowDays = 0;
  let consecutiveMisses = 0;
  let flowStartDate = today;
  let isBreathingToday = false;

  for (const date of sortedDates) {
    const log = findLog(logs, date);

    if (log?.completed) {
      flowDays++;
      consecutiveMisses = 0;
      flowStartDate = date;
    } else {
      consecutiveMisses++;
      if (consecutiveMisses >= 2) {
        break; // 흐름 끊김
      }
      if (date === today) {
        isBreathingToday = true;
      }
    }
  }

  return {
    currentFlowDays: flowDays,
    currentFlowStartDate: flowStartDate,
    isBreathingToday,
    longestFlow: calculateLongestFlow(logs),
    status: isBreathingToday ? 'breathing' :
            flowDays > 0 ? 'active' : 'broken'
  };
}
```

### 7-2. 최장 흐름 계산

```typescript
function calculateLongestFlow(logs: HabitLog[]): number {
  // 전체 기록을 처음부터 순회하며
  // 같은 흐름 규칙 적용 (연속 2일 미완료 = 끊김)
  // 모든 흐름 중 가장 긴 수행 일수 반환

  let longest = 0;
  let current = 0;
  let consecutiveMisses = 0;

  for (const date of getAllDatesAscending(logs)) {
    const log = findLog(logs, date);

    if (log?.completed) {
      current++;
      consecutiveMisses = 0;
      longest = Math.max(longest, current);
    } else {
      consecutiveMisses++;
      if (consecutiveMisses >= 2) {
        current = 0;
        consecutiveMisses = 0;
      }
    }
  }

  return longest;
}
```

### 7-3. 성능 고려사항

- **로컬 전용 계산**: 서버 불필요, AsyncStorage에서 읽어서 계산
- **캐싱**: 날짜가 바뀔 때만 재계산 (자정 기준)
- **최악 케이스**: 1년치 데이터 = 365개 로그 = 계산 시간 무시 가능
